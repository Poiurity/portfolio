Sellii

A resale-arbitrage agent, verified before it acts. 행동하기 전에, 서버가 먼저 검증하는 리셀 아비트리지 에이전트예요.

solo혼자 만듦 Mar–May 2026 not launched · private repo — code walkthrough available 미출시 · 비공개 저장소 — 코드 워크스루 가능

An agent built to scan Facebook Marketplace, OfferUp, and Craigslist on a daily schedule, price finds against eBay sold comps, and estimate net profit — a human approves every deal.

페이스북 마켓플레이스, OfferUp, 크레이그리스트를 매일 훑고, eBay 실거래가와 비교해 순이익을 추정하도록 만든 에이전트예요. 거래는 전부 사람이 승인해요.

The interesting part is trust. Instead of believing the model, the server recomputes every economic figure from canonical eBay comps and pins the evidence to each decision. I turned the reliability claims into three reproducible offline harnesses and published the result tables.

핵심은 신뢰예요. 모델이 말한 숫자를 믿는 대신, 서버가 eBay 실거래 데이터로 모든 금액을 다시 계산하고 근거를 결정에 박아둬요. 이 주장들을 오프라인에서 재현할 수 있는 하네스 3개로 직접 재서, 결과 표를 공개했어요.

CLIR

Scan a food label once — get a safe-or-not answer for your own dietary profile. 라벨을 한 번 스캔하면, 내 알레르기 프로필 기준으로 먹어도 되는지 바로 알려줘요.

● live · App Store앱스토어 서비스 중 backend owner, 4-person team4인 팀, 백엔드 담당 Mar 2026 – present 현재 App Store ↗ clir-app.com ↗

Every scan used to trigger a paid AI call for ingredient analysis — at realistic scale that breaks the cost model. I designed a perceptual-hash dedup cache, so a repeat label scan skips the model ladder entirely. Exact-match caching can't work here: two photos of the same label never hash identically.

원래는 스캔할 때마다 유료 AI 호출이 나가서, 규모가 커지면 비용 구조가 무너져요. 그래서 이미지 지각 해시로 중복을 잡는 캐시를 설계했어요. 같은 라벨을 다시 스캔하면 모델 호출을 아예 건너뛰어요. 같은 라벨이라도 사진마다 해시가 달라서, 단순 일치 캐시로는 안 되는 문제였어요.

The community feed survives request bursts with TTL jitter, stale-while-revalidate, and a distributed lock — N concurrent requests no longer trigger N recomputes of the 7-day aggregation.

커뮤니티 피드는 TTL 지터, stale-while-revalidate, 분산 락으로 지켰어요. 요청이 한꺼번에 몰려도 7일 집계 재계산이 N번 일어나지 않아요.

FlowCraft

A no-code Flutter app builder where agents never write the code directly. 에이전트가 코드를 직접 쓰지 않는, 노코드 Flutter 앱 빌더예요.

solo혼자 만듦 Mar 2026 – present 현재 github.com/Poiurity/FlowCraft ↗

Six LLM agents (Clarifier, WidgetExtender, Structure, Design, Repair, Critic) emit a typed AppState instead of raw Flutter; a deterministic generator compiles it to code. Generation runs in a closed loop: a 13-category static validator checks the output, a repair agent patches failures — two attempts max, with oscillation and wall-clock guards — and when repair fails, the system falls back to the prior valid state when one exists.

LLM 에이전트 6개(Clarifier, WidgetExtender, Structure, Design, Repair, Critic)가 Flutter 코드 대신 타입이 정해진 AppState를 만들고, 결정적 생성기가 그걸 코드로 바꿔요. 생성은 항상 닫힌 루프 안에서 돌아요. 13개 항목을 검사하는 정적 검증기가 결과를 확인하고, 실패하면 수리 에이전트가 고쳐요 — 최대 2번, 진동·시간 가드 포함. 그래도 안 되면 이전에 성공한 상태가 있을 때 거기로 돌아가요.

It started as an Orchestrator/Planning/UI/Logic pipeline; I redesigned it around typed state after raw code generation kept drifting between model calls. All 53 automated tests pass; preview runs in the browser via DartPad.

처음엔 Orchestrator/Planning/UI/Logic 파이프라인이었어요. 모델을 호출할 때마다 코드가 흔들려서, 타입 상태 중심으로 다시 설계했어요. 자동 테스트 53개가 전부 통과하고, 미리보기는 브라우저에서 DartPad로 돌아가요.

Stack — Flutter · Dart · OpenAI API · JSON Schema · DartPad

LiveTrack

solo개인 · Apr 2026– · GitHub ↗

Real-time location sharing for groups that split up on a trip. I wrote the WebSocket layer myself instead of reaching for a managed realtime service — the point was to own the failure handling: failures are classified by retryability, with separate paths for partially completed operations. The whole stack runs on one Hetzner VPS with Docker Compose; movement analytics run on PostGIS.

여행 중 흩어진 일행이 서로 위치를 실시간으로 보는 앱이에요. 관리형 실시간 서비스 대신 WebSocket 서버를 직접 짰어요 — 실패 처리를 스스로 감당해 보고 싶었거든요. 실패는 재시도 가능 여부로 나누고, 반쯤 끝난 작업은 따로 처리해요. 전체 스택은 Hetzner VPS 한 대에서 Docker Compose로 돌고, 이동 분석은 PostGIS로 해요.
